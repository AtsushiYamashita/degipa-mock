# degipa-mock プロジェクト要件書

**最終更新：2026年2月13日**

---

## プロジェクト概要

福祉医療の受援者にとって伴走者となるエージェントAIのアプリケーション

---

## 技術スタック

- **言語・フレームワーク：** TypeScript
- **データベース：** CozoDB
- **バックエンド基盤：** Firebase
- **ローカルDB：** wasm-cozodb（ブラウザ内）
- **リモートDB：** Firebase上のCozodb

---

## 主要機能

### 1. ユーザー向け機能

#### 日記機能
- ユーザーが日記を記述できる

#### 疑似SNS
- エージェントが演出する疑似SNS機能
- ユーザーが直接書き込み可能

#### 支援者に伝えることリスト
- エージェントがユーザーの日記・SNS投稿・相談メモの内容を元に自動生成
- エージェントの判断に基づいて重要度や優先度を付与
- ユーザーが読む専用リスト
- 支援者向けに共有可能な形式で出力

**相談メモとの関係性**
- **相談メモ**：ユーザーが主体的に入力・蓄積する「質問と回答」形式のデータ
  - 個人的で詳細
  - ユーザーのデバイスで暗号化されて保護される
  - 支援者には直接表示しない

- **支援者に伝えることリスト**：エージェントが相談メモ・日記・SNS投稿を分析して生成
  - 相談メモから「支援が必要な部分」を抽出・要約
  - 支援者に有用な形式に変換
  - より客観的・整理されたフォーマット
  - 支援者にセキュアに共有される

#### 相談メモ機能
- 画面下部にAIとの相談用テキストボックス
- ユーザーが「最近夜尿してることを追加」など気になることを入力

**作成フロー**
1. 相談メモ作成開始
2. エージェントが日付範囲を質問：「いつの日からいつの日までの出来事ですか？」
3. ユーザーが期間を指定
4. その期間を対象とした詳細なヒアリング開始
5. 掘り下げ質問を自動生成
   - 例）「どんなときに？」「周期は？」「その時の気分は？」など
   - ユーザーが応答するたびに相談メモが充実
6. 相談メモを閉じる際に「保存しますか？」を確認
   - 「保存」→ 以下参照の暗号化処理を実行
   - 「破棄」→ メモを削除

**セキュリティ仕様**
- 相談メモは個人情報として扱う
- 暗号化してからサーバーに保存
- 復号化はユーザーのデバイス上でのみ可能
- 鍵生成方式：ユーザーのMACアドレスを基に派生鍵を生成
  - デバイス固有であり、他のデバイスからのアクセスは不可
  - ローカルのwasm-cozodbに暗号化メモを保存
  - Firebaseに送信する際も暗号化状態で送信

---

### 2. エージェント機能

#### ローカル知識管理（wasm-cozodb）
- ブラウザ上で動作するCozodb
- ユーザーについての知識を蓄積・深める
- オフライン対応

#### リモート情報アクセス（Firebase上のCozodb）
- 地域の福祉情報にアクセス
- 適切な意思決定に必要な情報を取得

#### インテリジェント支援
- ユーザーの日記・SNS投稿を分析
- 地域福祉情報から適切なサポート情報を提供
- 「支援者に伝えることリスト」を生成

---

### 3. エージェント間連携機能（クローズド掲示板）

#### 仕様
- **ユーザー：** 書き込み不可（参照や間接的な恩恵のみ）
- **パートナーエージェント：** 読み書き可能

#### 用途
- 他のパートナーエージェント同士が知見を求める
- 例）「ウチのマスターが、薬を飲んだら動悸がして苦しいらしい、どうするといいかな？」
- 実例ベースの学習と改善

#### 育成コンセプト
- 自分の観察・外在化を通してパートナーエージェントを育てる
- パートナーエージェント同士が相互に学習
- その育ったパートナーから助けてもらう循環モデル

---

## データ構造・フロー概要

### ユーザーデータ
```
ユーザー入力（日記、SNS投稿）
  ↓
wasm-cozodb（ローカル知識蓄積）
  ↓
エージェント処理
  ├→ Firebase上Cozodb（福祉情報参照）
  └→ エージェント間掲示板への質問・回答
  ↓
支援者に伝えることリスト生成
  ↓
ユーザーに提示
```

---

## 実装の段階

1. **フロントエンド基盤** - ユーザーUI（日記、SNS、リスト表示）
2. **ローカルDB統合** - wasm-cozodb の組み込み
3. **エージェント基本機能** - LLM統合、基本的な分析
4. **バックエンド構築** - Firebase + Cozodb 連携
5. **エージェント間通信** - クローズド掲示板実装
6. **統合テスト・運用開始** - 全機能の統合検証

## エージェント設計ガイドライン：トラウマインフォームドケア

### 基本原則
エージェントがユーザーに何かを伝える・対応する際に、**トラウマインフォームドケアの専門家として**ワークフロー内で以下の自問チェックを実施。

### 自問チェックリスト

#### 1. 安全性（Safety）
- [ ] このメッセージはユーザーに危険感や不安感を与えないか？
- [ ] ユーザーが心理的に安全だと感じられる表現か？
- [ ] 予測不可能な言葉遣いや突然の指摘はないか？

#### 2. 信頼性と透明性（Trustworthiness & Transparency）
- [ ] 何をしようとしているのか、わかりやすく説明しているか？
- [ ] ユーザーが「なぜ？」と疑問に思うことなく理解できるか？
- [ ] 隠ぺいや曖昧な表現はないか？

#### 3. 選択と管理（Choice & Control）
- [ ] ユーザーが強制されていないか？拒否する選択肢を示しているか？
- [ ] 自分のペースで進めることができるようになっているか？
- [ ] 「やってみませんか？」という提案形式になっているか？

#### 4. 協働性（Collaboration）
- [ ] 一方的な判断・指示になっていないか？
- [ ] ユーザーの意見・希望・価値観を尊重しているか？
- [ ] 相談・対話のトーンになっているか？

#### 5. 力の回復（Empowerment）
- [ ] ユーザーの強みや資源に焦点を当てているか？
- [ ] 受身的ではなく、主体的な行動を支援しているか？
- [ ] ユーザーの自己決定能力を尊重しているか？

### 実装方法
- エージェントが回答を生成する前に、上記チェックを内部で実行
- チェック結果に基づいて回答を調整・修正
- ユーザーには見えない バックエンドプロセス として動作
- ログに記録されて、エージェント改善の参考データとなる

## 重要な設計ポイント

- **ユーザープライバシー・セキュリティ**
  - 相談メモやその他個人情報は暗号化して保存
  - MACアドレスベースの派生鍵でデバイス固有の復号化を実現
  - ユーザーのデバイス上でのみ復号化可能
- **データ分離**：ローカルデータ（相談メモ）とリモートデータ（福祉情報）の適切な分離
- **エージェント間の学習**：無限ループ防止と信頼性・精度維持の仕組み
- **ユーザー体験**：エージェントとの自然な関係構築
- **スケーラビリティ**：複数エージェント、複数ユーザーの管理
- **倫理的フレームワーク**：トラウマインフォームドケアが全ての応答に組み込まれる
