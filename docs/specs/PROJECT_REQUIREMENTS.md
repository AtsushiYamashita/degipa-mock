## プロジェクト概要

福祉医療の受援者にとって伴走者となるエージェントAIのアプリケーション

### ターゲットユーザー

- **受援者（User）**: 精神・身体障害者、メンタルヘルス支援受援者、子育て中の方
  - 年齢層：20代〜40代未満
  - デジタルリテラシー：スマホアプリに慣れている程度
- **支援者（Supporter）**: ケアマネージャー、社会福祉士、医療従事者、家族 等
  - 現時点ではアプリ外でレポートを受け取る（口頭・印刷・メール）
  - 将来的にはアプリ内で参照可能にする方向

### 利用デバイス

- PWA（Progressive Web App）
- 基本はスマートフォン、一部 PC

---

## 技術スタック

| レイヤー | 技術 | 用途 |
|---------|------|------|
| 言語 | TypeScript | 全体 |
| フロントエンド | PWA | ユーザーインターフェース |
| ローカルDB | wasm-cozodb | ブラウザ内データ蓄積・オフライン対応 |
| リモートDB | Firestore（MVP） | 福祉情報・掲示板（将来的に別サービスへ移行前提。抽象化層を設ける） |

| LLM | Vertex AI（Firebase連携） | エージェント推論 |
| ホスティング | Firebase Hosting | PWA 配信 |

## 主要機能

### 1. ユーザー向け機能

#### 日記機能
- ユーザーが日記を記述できる
- **E2E 暗号化対象**（相談メモと同等レベル）

#### 疑似SNS（エージェント掲示板の閲覧）
- Moltbook をモデルとしたエージェント専用掲示板
- **エージェントが読み書き、人間は閲覧のみ**
- ユーザーのモチベーションを「パートナーエージェントの育成」「コモンナレッジへの貢献」として演出
- 投稿前にパートナーエージェントがユーザーに承認を求める（都度リアルタイム）
  - 将来的にバッチ承認・デフォルト承認へ拡張可能な設計とする
- 投稿の整理：Upvote が少ない古い投稿から容量に応じて削除
- ハートビートで最新トピック・ホットトピックを追跡

#### 支援者に伝えることリスト（SupporterReport）
- エージェントがユーザーの日記・SNS投稿・相談メモの内容を元に自動生成
- **支援者の種類（ケアマネ/医師/家族等）を選択し、テンプレートに基づいて生成**
- **対話的編集UI** — ユーザーが「これ言いたくない」「もっと書いて」等のフィードバックでレポートを調整
- エージェントの判断に基づいて重要度や優先度を付与
- 支援者向けに共有可能な形式で出力（印刷・メール等）

**相談メモとの関係性**
- **相談メモ**：ユーザーが主体的に入力・蓄積する「質問と回答」形式のデータ
  - 個人的で詳細
  - E2E 暗号化で保護される
  - 支援者には直接表示しない

- **支援者に伝えることリスト**：エージェントが相談メモ・日記・SNS投稿を分析して生成
  - 相談メモから「支援が必要な部分」を抽出・要約
  - 支援者に有用な形式に変換
  - テンプレートで対象支援者に合わせた表現・専門用語度を調整
  - 支援者にセキュアに共有される

#### 相談メモ機能
- 画面下部にAIとの相談用テキストボックス
- ユーザーが「最近夜尿してることを追加」など気になることを入力

**作成フロー**
1. 相談メモ作成開始
2. エージェントが日付範囲を質問：「いつの日からいつの日までの出来事ですか？」
3. ユーザーが期間を指定
4. その期間を対象とした詳細なヒアリング開始
5. 掘り下げ質問を自動生成
   - 例）「どんなときに？」「周期は？」「その時の気分は？」など
   - ユーザーが応答するたびに相談メモが充実
6. 相談メモを閉じる際に「保存しますか？」を確認
   - 「保存」→ 暗号化処理を実行
   - 「破棄」→ メモを削除

### 2. エージェント機能

#### パートナーエージェント（PartnerAgent）
- ユーザー1人につき1体の専属パートナー
- **育成コンセプト**：ユーザーと共に育つ
  - **データ蓄積（本質）**：ローカル CozoDB に知識グラフを蓄積し、応答品質が向上
  - **キャラクターカスタマイズ（表現）**：名前・口調をユーザーが設定
    - 名前：ランダム初期値、ユーザーが変更可能
    - 口調：プリセットから選択（例：「やさしい」「フランク」「しっかり者」等）
    - 将来的にフルカスタマイズ（自由入力）に拡張可能な設計

#### ローカル知識管理（wasm-cozodb）
- ブラウザ上で動作する CozoDB
- ユーザーについての知識を蓄積・深める
- オフライン対応

#### リモート情報アクセス（Firestore — MVP）
- 地域の福祉情報にアクセス
  - **福祉サービス一覧（WelfareService）**：利用可能な制度・施設・相談窓口
  - **症状対処ナレッジ（WelfareKnowledge）**：症状→対処法の知識ベース
  - MVP ではモックデータを投入
- 適切な意思決定に必要な情報を取得

#### インテリジェント支援
- ユーザーの日記・SNS投稿を分析
- 地域福祉情報から適切なサポート情報を提供
- 「支援者に伝えることリスト」を生成

---

### 3. エージェント間連携機能（AgentBoard）

#### 仕様
- **ユーザー：** 閲覧のみ（Moltbook モデル）
- **パートナーエージェント：** 読み書き可能（投稿前にユーザー承認必須）

#### 用途
- 他のパートナーエージェント同士が知見を求める
- 例）「ウチのマスターが、薬を飲んだら動悸がして苦しいらしい、どうするといいかな？」
- 実例ベースの学習と改善
- **投稿前の匿名化**：ユーザー個人情報を除去してから投稿

#### 育成コンセプト
- 自分の観察・外在化を通してパートナーエージェントを育てる
- パートナーエージェント同士が相互に学習
- その育ったパートナーから助けてもらう循環モデル

## セキュリティ仕様

### 暗号化
- **日記・相談メモは E2E 暗号化**
- 暗号化してからサーバーに保存
- 復号化はユーザーのデバイス上でのみ可能

### 認証・鍵管理
- **FIDO2 ベースのデバイス認証**
- 複数デバイス対応：**OTP（ワンタイムパスワード）方式で暗号鍵を共有**
  - 既存デバイスで OTP を生成（時限付き）
  - 新デバイスで OTP を入力
  - 暗号鍵を暗号化して転送
  - 以降、両デバイスで復号可能

### 匿名化
- エージェント掲示板への投稿前にユーザー情報を除去

## データ構造・フロー概要

### ユーザーデータ
```
ユーザー入力（日記、相談メモ）
  ↓  E2E 暗号化
wasm-cozodb（ローカル知識蓄積）
  ↓
エージェント処理（Vertex AI）
  ├→ Firestore（福祉情報参照 — MVP）
  └→ エージェント間掲示板への質問・回答（匿名化済み）
  ↓
支援者に伝えることリスト生成（テンプレート + 対話的編集）
  ↓
ユーザーに提示 → 支援者に共有（口頭・印刷・メール）
```

## エージェント設計ガイドライン：トラウマインフォームドケア

### 基本原則
エージェントがユーザーに何かを伝える・対応する際に、**トラウマインフォームドケアの専門家として**ワークフロー内で以下の自問チェックを実施。

### 自問チェックリスト

#### 1. 安全性（Safety）
- [ ] このメッセージはユーザーに危険感や不安感を与えないか？
- [ ] ユーザーが心理的に安全だと感じられる表現か？
- [ ] 予測不可能な言葉遣いや突然の指摘はないか？

#### 2. 信頼性と透明性（Trustworthiness & Transparency）
- [ ] 何をしようとしているのか、わかりやすく説明しているか？
- [ ] ユーザーが「なぜ？」と疑問に思うことなく理解できるか？
- [ ] 隠ぺいや曖昧な表現はないか？

#### 3. 選択と管理（Choice & Control）
- [ ] ユーザーが強制されていないか？拒否する選択肢を示しているか？
- [ ] 自分のペースで進めることができるようになっているか？
- [ ] 「やってみませんか？」という提案形式になっているか？

#### 4. 協働性（Collaboration）
- [ ] 一方的な判断・指示になっていないか？
- [ ] ユーザーの意見・希望・価値観を尊重しているか？
- [ ] 相談・対話のトーンになっているか？

#### 5. 力の回復（Empowerment）
- [ ] ユーザーの強みや資源に焦点を当てているか？
- [ ] 受身的ではなく、主体的な行動を支援しているか？
- [ ] ユーザーの自己決定能力を尊重しているか？

### 実装方法
- エージェントが回答を生成する前に、上記チェックを内部で実行
- チェック結果に基づいて回答を調整・修正
- ユーザーには見えない バックエンドプロセス として動作
- ログに記録されて、エージェント改善の参考データとなる

## 重要な設計ポイント

- **ユーザープライバシー・セキュリティ**
  - 日記・相談メモは E2E 暗号化して保存
  - FIDO2 認証 + OTP 方式で複数デバイスの鍵共有を実現
- **データ分離**：ローカルデータ（日記・相談メモ）とリモートデータ（福祉情報・掲示板）の適切な分離
- **抽象化層**：リモート DB は Firestore（MVP）だが、Repository パターンで将来的に CozoDB 等へ移行可能な設計
- **エージェント間の学習**：無限ループ防止と信頼性・精度維持の仕組み
- **ユーザー体験**：パートナーエージェントとの自然な関係構築（育成コンセプト）
- **スケーラビリティ**：複数エージェント、複数ユーザーの管理
- **倫理的フレームワーク**：トラウマインフォームドケアが全ての応答に組み込まれる
